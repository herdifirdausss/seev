# Threat Model — Seev Internal Service Plane

> [Documentation home](../README.md) · [Security](README.md)

> **Status: Current. Audience: security reviewers, maintainers, and
> operators.** A threat model is a map of what must be protected, who is
> trusted at each boundary, and what can go wrong. It is not proof that a
> deployment is secure.

Last reviewed: 2026-07-22, after completion of
[plan 49](../roadmap/archive/49-a6-internal-security.md).

This living document describes the current security boundaries of the local
reference system. It is not a production certification or a claim that the
default Compose configuration is safe for real customer data. Plan 49 retains
the detailed implementation and test history behind the controls summarized
here.

## 1. Scope

This model covers:

- public HTTP entry points at Gateway and Auth;
- gRPC and internal HTTP traffic among the eight deployable services;
- the Admin BFF operator boundary;
- service access to PostgreSQL, Redis, RabbitMQ, and the optional local Vault;
- observability access to service metrics; and
- secrets and certificates used by the local Compose topology.

It does not model cloud perimeter controls, production TLS termination,
managed secret stores, employee devices, real vendor infrastructure, or legal
and regulatory certification. Those require deployment-specific decisions
that this learning repository deliberately does not make.

## 2. Security posture at a glance

In plain English, an internal service is not trusted merely because it is on
the same network. It must prove its identity, be explicitly allowed by the
receiver, and still pass the business authorization for the requested action.
The local public edge is intentionally incomplete: a real deployment must add
its own TLS termination and perimeter controls.

- Every internal gRPC and internal HTTP connection uses mutual TLS.
- A certificate's SPIFFE-style URI SAN is the service identity. A certificate
  signed by the local CA is not sufficient by itself; the caller must also be
  on the callee's explicit allowlist.
- Internal gRPC also requires a non-empty shared token. It is defense in depth,
  not the service identity mechanism.
- Admin HTTP requests require both an allowed mTLS identity and the applicable
  JWT role. Ledger enforces maker/checker rules in the owning service.
- `/metrics` is served on internal listeners and scraped with the dedicated
  Prometheus mTLS identity.
- Certificates are short-lived and hot-reloaded. Rotation has a bounded poll
  interval during which a newly issued client may be rejected temporarily.
- Each service owns one PostgreSQL database and uses a restricted application
  role. Cross-service database access is forbidden.
- Local defaults and mock integrations remain development conveniences. Real
  deployments must replace them and provide edge TLS and production-grade
  secret management.

## 3. Assets

| Asset | Owner/location | Primary impact if compromised |
|---|---|---|
| Ledger balances and entries | Ledger / `seev_ledger` | Unauthorized, duplicated, or concealed money movement |
| User identity and KYC data | Auth / `seev_auth` | Account takeover, privacy breach, compliance exposure |
| JWT signing secret | Shared service configuration | Forged user or administrator identity across services |
| Internal gRPC token | Shared service configuration | Loss of one defense-in-depth control on RPC calls |
| Service certificates and local CA | `deploy/certs`, generated by `cmd/certgen` | Service impersonation inside the local network |
| Database credentials | Per-service configuration | Direct access to one service's owned data |
| Vendor webhook and API secrets | Payin/Payout configuration | Forged vendor callbacks or commands |
| Admin sessions and audit log | Admin BFF / `seev_adminbff` | Operator takeover or loss of action traceability |
| Screening and assurance findings | Fraud and Assurance databases | Hidden risk events or false confidence in product state |

## 4. Trust boundaries

### Public edge

Gateway (`:8080`) and Auth (`:8082`) expose the end-user API. Users
authenticate with JWTs; vendor webhooks use HMAC signatures. Public health and
readiness probes are unauthenticated by design. The local stack uses plain
HTTP at this boundary, so any real deployment must add trusted TLS termination
and a deployment-specific proxy policy.

### Internal gRPC

There are thirteen current RPC call paths among Gateway, Auth, Ledger, Payin,
Payout, Fraud, and Assurance. Each uses mTLS, a caller allowlist, and the
required internal token. Admin BFF has no gRPC listener.

The security boundary is evaluated in this order:

1. validate the certificate chain against the hot-reloaded CA;
2. extract and validate the caller's URI SAN identity;
3. verify that the callee allows that identity; and
4. validate the internal token and request-level business authorization.

### Internal HTTP and operator access

Internal/admin listeners are loopback-published for local tooling and are also
reachable on the Compose network. mTLS rejects callers before HTTP routing.
Admin BFF uses server-side sessions and CSRF protection for the browser,
then sends short-lived operator JWTs to six downstream admin APIs. Assurance
is operated through its dedicated admin API or `scripts/product-assurance.sh`.

### Data stores and messaging

Services use separate databases and restricted roles in one local PostgreSQL
container. RLS and grants reduce accidental cross-tenant and cross-role
access. Redis and RabbitMQ are infrastructure dependencies on the local
network; transport security for those dependencies is deployment work, not a
claim made by this repository.

### Secrets and observability

Environment variables remain the default local secret source. The optional
Vault profile demonstrates per-key secret lookup and fallback, but Vault is
not mandatory and its development listener uses plain HTTP. Prometheus uses a
dedicated certificate identity to scrape internal `/metrics` endpoints.

## 5. Service topology

| Service | Public role | Important internal role |
|---|---|---|
| Gateway | End-user API and vendor webhook edge | gRPC client, ledger HTTP proxy, notification consumer |
| Auth | Registration, login, profile, and KYC API | Ledger/Fraud client and KYC workers |
| Ledger | Public API only through Gateway | Posting source of truth, RPC server, admin API, workers |
| Payin | None | Top-up RPC server, Ledger/Fraud client, admin API |
| Payout | None | Payout RPC server, Ledger/Fraud client, vendor relay, admin API |
| Fraud | None | Screening RPC server, event consumer, admin API |
| Admin BFF | Local operator browser surface | Session boundary and typed HTTP proxy |
| Assurance | None | Read-only correlation client, findings and intake-control API |

`cmd/sanctions-loader`, `cmd/certgen`, and `cmd/gentoken` are batch or local
utilities, not network services.

## 6. Findings register

Status meanings:

- **Resolved** — the current tree contains the control and verification named
  in the evidence column.
- **Partially addressed** — a safer option exists, but the insecure local
  default remains available.
- **Accepted risk** — the limitation is deliberate for the local learning
  scope and has an explicit production follow-up.

| ID | Finding | Severity | Status | Evidence/current decision |
|---|---|---:|---|---|
| TM-01 | Empty internal token disabled RPC authentication. | Critical | Resolved | gRPC server and dial helpers reject an empty token; covered by `TestNewServer_EmptyTokenFailsFast`. |
| TM-02 | Internal gRPC and HTTP used plaintext transport without service identity. | Critical | Resolved | `pkg/tlsx`, per-hop URI SAN allowlists, and all eight service configurations enforce mTLS. |
| TM-03 | A valid admin JWT could reach internal APIs without proving which service sent it. | High | Resolved | mTLS identity is checked before JWT and role authorization. |
| TM-04 | Metrics were unauthenticated and scraped over plaintext HTTP. | Medium | Resolved | Internal metrics listeners require mTLS and Prometheus uses its own allowed identity. |
| TM-05 | Application secrets default to plaintext environment configuration. | High | Partially addressed | Optional Vault loading is available, but production enforcement is intentionally outside this repo. |
| TM-06 | Default CORS allowed every origin. | Medium | Resolved | API-only default is empty; non-production origins require an explicit `CORS_ALLOWED_ORIGINS` value. |
| TM-07 | JWT issuer validation could be disabled by an empty configuration. | Medium | Resolved | Every service now requires `JWT_ISSUER`; generated test tokens include the issuer claim. |
| TM-08 | The mock vendor format has no independent signed delivery timestamp. | Low | Accepted risk | Full-body HMAC and event-ID dedup protect the test flow. A real vendor integration must implement that vendor's replay and clock-skew contract. |
| TM-09 | Assurance was absent from the initial identity and metrics inventory. | Medium | Resolved | Assurance has its own certificate, three protected RPC dials, an mTLS HTTP listener, and a Prometheus target. |
| TM-10 | The optional development Vault serves secrets over plain HTTP. | Medium | Accepted risk | The profile is local-only. A production secret store and transport policy are deployment requirements. |
| TM-11 | Rate-limit keys included the ephemeral client port, allowing new connections to bypass one IP bucket. | Medium | Resolved | Rate-limit helpers strip the source port and ignore client-supplied forwarding headers; unit and chaos tests cover the behavior. |
| TM-12 | Request logging restored handlers from a 16 KiB truncated copy. | Medium-High | Resolved | The logger restores the complete body up to the 10 MiB repository safety cap and separately truncates only the log copy. |
| TM-13 | Server TLS configuration held a frozen CA pool after certificate rotation. | High | Resolved | Each handshake reads the current CA pool. Rotation tests prove new certificates recover without restart and old certificates remain rejected. |
| TM-14 | A black-holed Redis velocity request could consume the caller deadline and be misclassified as fail-open. | High | Resolved | Fraud's Redis attempt has a 150 ms sub-deadline, preserving time to return `DEPENDENCY_UNAVAILABLE` and fail closed before money movement. |

The detailed reproduction steps, design decisions, and final isolated gate for
TM-01 through TM-14 are recorded in
[plan 49](../roadmap/archive/49-a6-internal-security.md). Operational procedures live in
the [runbook index](../operations/runbooks/README.md).

## 7. Review requirements

Update this document in the same change whenever any of these occurs:

- a service, listener, RPC call, or internal HTTP call is added;
- an mTLS allowlist or certificate identity changes;
- a new secret source or credential is introduced;
- an endpoint moves between public and internal exposure;
- a finding changes status; or
- a production deployment model is added to the repository.

Do not mark a finding resolved from code review alone. Record the test,
integration check, drill, or live rejection that proves the control works.
